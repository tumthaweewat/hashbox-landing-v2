ธุรกิจไทยเกือบทุกเจ้ามี LINE OA อยู่แล้ว แต่ส่วนใหญ่ยังตั้ง auto-reply แบบ keyword หรือ rich menu ที่ตอบได้แค่คำถามที่เดาไว้ล่วงหน้า พอลูกค้าพิมพ์นอกสคริปต์ก็เงียบ หรือเด้งไปหาแอดมินทันที การเพิ่ม AI เข้าไปบน LINE OA ที่มีอยู่คือ upgrade ที่ลงทุนน้อยแต่เห็นผลเร็ว เพราะไม่ต้องย้ายลูกค้าไปแพลตฟอร์มใหม่ ใช้ช่องทางที่ลูกค้าคุ้นอยู่แล้ว บทความนี้อธิบายว่า `line chatbot ai` ทำอะไรได้จริง ต่างจากของเดิมยังไง ต้องเตรียมอะไร ราคาเท่าไร และข้อควรระวังที่ต้องรู้ก่อนเริ่ม

## LINE Chatbot AI ทำอะไรได้บ้าง {#features}

`แชทบอท ai line` ที่ทำมาดีจะรับงานหน้าบ้านแทนแอดมินได้หลายอย่าง โดยเฉพาะงานตอบซ้ำ ๆ ที่กินเวลาทีมทุกวัน:

- **ตอบคำถามลูกค้า 24/7** — ราคา โปรโมชั่น เวลาทำการ วิธีสั่งซื้อ เงื่อนไขจัดส่ง ลูกค้าทักตอนตี 2 ก็ได้คำตอบทันที ไม่ต้องรอแอดมินเปิดร้าน
- **Qualify และเก็บ lead** — ถามคำถามคัดกรอง (งบประมาณ ความต้องการ ช่วงเวลา) แล้วบันทึกข้อมูลลูกค้าที่พร้อมซื้อ ส่งเข้า CRM หรือ Google Sheet ให้ทีมขายตามต่อ
- **จองคิว / นัดหมาย** — เช็คช่วงเวลาว่าง จองคิวคลินิก ร้านทำผม หรือนัดปรึกษา แล้วส่งข้อความยืนยันกลับอัตโนมัติ
- **ตอบจากเอกสารบริษัทด้วย RAG** — ดึงคำตอบจาก catalog สินค้า คู่มือ ราคากลาง หรือ FAQ ภายในที่คุณอัปโหลดไว้ ทำให้บอทตอบได้ "ตรงกับข้อมูลจริงของร้าน" ไม่ใช่มั่วขึ้นมาเอง
- **ส่งต่อแอดมินเมื่อจำเป็น** — ถ้าเจอคำถามที่ตอบไม่ได้ ลูกค้าโมโห หรือเป็นดีลใหญ่ บอทจะโบกมือเรียกคนจริงเข้ามาแทน พร้อมสรุปบทสนทนาให้แอดมินอ่านก่อน

สรุปสั้น ๆ คือ AI รับงานตอบซ้ำ 60-80% ให้ ปล่อยให้ทีมโฟกัสเคสที่ต้องใช้คนจริง แนวคิดเดียวกับที่เราเขียนไว้ใน[บทความ AI ช่วยลดงาน manual และประหยัดเวลาทีม 20 ชั่วโมงต่อสัปดาห์](/5-ways-ai-save-team-20-hours-week/) เพียงแต่ย้ายมาอยู่บนช่องทางที่ลูกค้าไทยใช้ทุกวันอย่าง LINE

## ต่างจาก rule-based / keyword bot เดิมยังไง {#vs-rulebased}

หลายคนสับสนว่า LINE มี auto-reply อยู่แล้ว จะทำ AI ไปทำไม ความต่างสำคัญคือ auto-reply เดิมเป็น **rule-based** (ตอบตามกฎที่ตั้งไว้) ส่วน AI chatbot **เข้าใจความหมาย** ของสิ่งที่ลูกค้าพิมพ์ ตารางนี้เทียบให้เห็นชัด:

| หัวข้อ | Rule-based / Keyword bot เดิม | LINE Chatbot AI |
|---|---|---|
| วิธีเข้าใจคำถาม | จับ keyword ตรงตัว เช่น ต้องพิมพ์ว่า "ราคา" เป๊ะ | เข้าใจความหมาย พิมพ์ว่า "เท่าไหร่", "กี่บาท", "แพงไหม" ก็เข้าใจ |
| คำถามนอกสคริปต์ | ตอบไม่ได้ เด้ง default message | พยายามตอบจากข้อมูลที่มี หรือส่งต่อแอดมิน |
| บทสนทนาต่อเนื่อง | จำ context ไม่ได้ เริ่มใหม่ทุกข้อความ | จำบริบทได้ ถามต่อเนื่องเป็นเรื่องเดียวกันได้ |
| แหล่งคำตอบ | คำตอบที่พิมพ์ hard-code ไว้ | ดึงจากเอกสารบริษัทจริงผ่าน RAG |
| การดูแล | ต้องมานั่งเพิ่ม keyword เองเรื่อย ๆ | อัปเดตเอกสาร knowledge base ก็พอ |
| ภาษา | ตอบตามที่เขียนไว้ | ตอบภาษาไทย-อังกฤษปนได้ ปรับโทนได้ |
| ต้นทุน | ฟรี/ถูก แต่ครอบคลุมแคบ | มีค่า token แต่ตอบได้กว้างและเหมือนคนกว่ามาก |

จุดที่ทำให้ `line oa ai` คุ้มค่าคือ ลูกค้าจริงไม่พิมพ์ตามที่เราเดา เขาพิมพ์สั้น พิมพ์ผิด ใช้คำแสลง หรือถามหลายอย่างในประโยคเดียว rule-based bot จะตกม้าตายตรงนี้ ส่วน AI จับใจความได้และตอบได้ตรงกว่า

## สถาปัตยกรรมแบบเข้าใจง่าย {#architecture}

`chatbot ai ธุรกิจไทย` ที่ตอบจากข้อมูลบริษัทได้ ประกอบด้วย 3 ส่วนหลักต่อกันเป็นสายพาน ไม่ซับซ้อนอย่างที่คิด:

```
ลูกค้าพิมพ์ใน LINE
        │
        ▼
[1] LINE OA (Messaging API)  ← รับ-ส่งข้อความ
        │  webhook
        ▼
[2] Backend (ระบบเรา)  ← ตัวกลาง จัดคิว/ตรรกะ/บันทึก log
        │
        ├──► [3a] RAG: ค้นข้อมูลที่เกี่ยวข้อง
        │         จาก knowledge base (เอกสารบริษัท)
        │
        └──► [3b] LLM: เอาคำถาม + ข้อมูลที่ค้นได้
                  มาเรียบเรียงเป็นคำตอบภาษาธรรมชาติ
        │
        ▼
ส่งคำตอบกลับเข้า LINE ให้ลูกค้า
```

อธิบายทีละส่วน:

1. **LINE OA (Messaging API)** — เป็นประตูหน้าบ้าน เมื่อลูกค้าพิมพ์ LINE จะยิง webhook มาที่ระบบเรา และเป็นช่องทางที่เราส่งคำตอบกลับ ต้องใช้ LINE OA แบบที่เปิด Messaging API ได้
2. **Backend** — เป็นสมองส่วนกลาง คอยจัดการว่าคำถามนี้ควรให้ AI ตอบ ควรส่งต่อแอดมิน หรือควรบันทึกเป็น lead รวมถึงเก็บ log และควบคุม guardrail
3. **LLM + RAG** — หัวใจของความฉลาด **RAG (Retrieval-Augmented Generation)** จะไปค้นเอกสารบริษัทที่เกี่ยวข้องกับคำถามก่อน แล้วส่งข้อมูลนั้นให้ **LLM** เรียบเรียงเป็นคำตอบ วิธีนี้ทำให้บอทตอบ "อิงข้อมูลจริง" ไม่ใช่เดาจากความรู้ทั่วไปของโมเดล ซึ่งเป็นกุญแจสำคัญในการลดการมั่ว (hallucination)

ที่ Hashbox เราวางระบบ RAG, Sales GPT และ workflow automation แบบนี้เป็นบริการหลักอยู่แล้ว จุดที่ต้องออกแบบให้ดีคือชั้น RAG และ guardrail เพราะเป็นตัวตัดสินว่าบอทจะตอบแม่นหรือมั่ว หากอยากเข้าใจภาพรวมการวางระบบตั้งแต่ต้น อ่านเพิ่มได้ที่[บริการที่ปรึกษา AI สำหรับธุรกิจ](/services/ai-consulting/)

## ต้องเตรียมอะไรก่อนเริ่ม {#prepare}

ก่อนจะ `ทำ chatbot line` ให้ฉลาด ของที่ต้องเตรียมไม่ได้เยอะ แต่ต้องมีให้ครบ ไม่อย่างนั้น AI จะไม่มีข้อมูลให้ตอบ:

1. **LINE OA account ที่เปิด Messaging API ได้** — ถ้ามี LINE OA อยู่แล้วก็ใช้ต่อได้เลย แค่เปิดสิทธิ์ Messaging API ในระบบ
2. **เอกสาร / FAQ / ข้อมูลสินค้า** — catalog สินค้า ตารางราคา เงื่อนไขจัดส่ง คำถามที่ลูกค้าถามบ่อย รวบรวมเป็นไฟล์ (PDF, Excel, Google Doc ก็ได้) นี่คือ "สมอง" ที่ RAG จะไปดึงมาตอบ ยิ่งข้อมูลชัดและอัปเดต บอทยิ่งตอบแม่น
3. **Use-case ที่ชัดเจน** — เลือกก่อนว่าจะให้บอทเน้นอะไร ตอบคำถามสินค้า? จองคิว? เก็บ lead? อย่าพยายามให้ทำทุกอย่างพร้อมกันตั้งแต่วันแรก เริ่มจาก 1-2 use-case ที่กินเวลาทีมมากสุด
4. **งบประมาณและคนดูแล** — ตั้งงบ setup และค่า token รายเดือน พร้อมกำหนดคนในทีมที่จะคอยดู log ปรับ knowledge base และรับเคสที่บอทส่งต่อ

เคล็ดลับคือ เริ่มจากรวบรวมคำถามจริงที่ลูกค้าเคยทักเข้ามาใน LINE ย้อนหลัง 1-2 เดือน แล้วจัดกลุ่ม จะเห็นชัดว่าคำถาม 20% ที่ถูกถามบ่อยสุดกินสัดส่วนแชทส่วนใหญ่ ให้เอา 20% นั้นมาทำก่อน

## ข้อควรระวังก่อนทำ {#caveats}

AI chatbot ไม่ใช่ของวิเศษที่เสียบแล้วจบ มี 3 เรื่องที่ต้องเข้าใจตั้งแต่ต้นเพื่อไม่ให้เจอปัญหาทีหลัง:

- **Hallucination (การมั่ว)** — LLM มีโอกาสตอบมั่นใจแต่ผิด โดยเฉพาะเรื่องราคาหรือเงื่อนไข วิธีคุมคือ **grounding ด้วย RAG** (บังคับให้ตอบจากเอกสารจริงเท่านั้น) บวกกับ **guardrails** เช่น ถ้าไม่มีข้อมูลให้บอกว่าไม่ทราบและส่งต่อแอดมิน ห้ามเดา และควรตั้ง disclaimer ในเรื่องอ่อนไหว เช่น ราคา ให้ยืนยันกับทีมขายอีกครั้ง
- **PDPA / ข้อมูลส่วนบุคคล** — บอทเก็บเบอร์ ชื่อ ที่อยู่ลูกค้า ต้องมีการแจ้งขอความยินยอม เก็บข้อมูลเท่าที่จำเป็น กำหนดสิทธิ์การเข้าถึง และมีนโยบายลบข้อมูล ให้สอดคล้อง PDPA เรื่องนี้ต้องออกแบบตั้งแต่ต้น ไม่ใช่มาแก้ทีหลัง
- **ต้นทุน token** — ทุกข้อความที่ AI ตอบมีต้นทุน ยิ่งบทสนทนายาวหรือ knowledge base ใหญ่ ยิ่งใช้ token มาก ควรออกแบบให้ดึงเฉพาะข้อมูลที่เกี่ยวข้อง (retrieval ที่ดี) และตั้งเพดานการใช้งาน เพื่อไม่ให้ค่าใช้จ่ายบานปลาย

ข้อควรระวังพวกนี้ไม่ได้แปลว่าไม่ควรทำ แต่แปลว่าควรทำกับคนที่ออกแบบ RAG และ guardrail เป็น เพราะความต่างระหว่างบอทที่ลูกค้าไว้ใจกับบอทที่สร้างปัญหา อยู่ที่ชั้นเหล่านี้ล้วน ๆ

## FAQ {#faq}

### LINE Chatbot AI ต่างจาก LINE auto-reply เดิมยังไง?
Auto-reply เดิมเป็น rule-based ตอบตาม keyword ที่ตั้งไว้ พิมพ์นอกสคริปต์ก็ตอบไม่ได้ ส่วน AI chatbot เข้าใจความหมายของคำถาม จำบทสนทนาได้ และดึงคำตอบจากเอกสารบริษัทจริงผ่าน RAG ตอบได้กว้างและเหมือนคนคุยมากกว่า

### ต้องมี LINE OA แบบเสียเงินไหม?
ใช้ LINE OA ที่มีอยู่ต่อได้ ขอแค่เปิดสิทธิ์ Messaging API ได้ ค่าใช้จ่ายฝั่ง LINE ขึ้นกับแพ็กเกจข้อความที่คุณใช้อยู่ ส่วนค่า AI (setup + token) เป็นคนละก้อนกับค่า LINE

### AI ตอบผิดแล้วทำยังไง?
คุมด้วย 3 ชั้น: (1) RAG บังคับให้ตอบจากเอกสารจริงเท่านั้น (2) guardrail ให้บอกว่าไม่ทราบและส่งต่อแอดมินเมื่อไม่มีข้อมูล ห้ามเดา (3) ทีมคอยดู log แล้วปรับ knowledge base เรื่องอ่อนไหวอย่างราคาควรตั้งให้ยืนยันกับทีมขายอีกครั้ง

### ใช้เวลาทำนานแค่ไหน?
ระดับ Starter (ตอบ FAQ + ข้อมูลสินค้า + ส่งต่อแอดมิน) ประมาณ 2-3 สัปดาห์ ถ้าเพิ่มการเก็บ lead จองคิว หรือต่อระบบหลังบ้าน จะใช้ราว 4-6 สัปดาห์ขึ้นไป ขึ้นกับความพร้อมของเอกสารและระบบที่ต้องเชื่อม

### ราคาเท่าไร?
เริ่ม 60,000 บาทสำหรับระดับ Starter บวกค่า token รายเดือนตามปริมาณการใช้จริง และค่าดูแล knowledge base เป็นระยะ ระดับที่ซับซ้อนขึ้นประเมินตาม scope แนะนำให้เริ่มจาก audit use-case ก่อนเพื่อดูจุดที่คุ้มสุด

### ข้อมูลลูกค้าปลอดภัยไหม เรื่อง PDPA?
ต้องออกแบบให้สอดคล้อง PDPA ตั้งแต่ต้น: แจ้งขอความยินยอมก่อนเก็บข้อมูล เก็บเท่าที่จำเป็น กำหนดสิทธิ์การเข้าถึง และมีนโยบายลบข้อมูล เรื่องนี้เป็นส่วนหนึ่งของการวางระบบที่ดี ไม่ใช่ของแถมที่ค่อยมาแก้ทีหลัง

---

**สรุป:** LINE Chatbot AI คือการเพิ่ม LLM + RAG เข้าไปบน LINE OA ที่ธุรกิจไทยมีอยู่แล้ว เพื่อให้ตอบลูกค้าเป็นภาษาธรรมชาติจากข้อมูลบริษัทจริง ต่างจาก auto-reply เดิมที่ตอบตาม keyword ตายตัว โดยรับงานตอบซ้ำ ๆ ได้ 24/7 qualify lead จองคิว และส่งต่อแอดมินเมื่อจำเป็น สถาปัตยกรรมมี 3 ส่วนคือ LINE OA + backend + LLM/RAG ก่อนเริ่มต้องเตรียม LINE OA ที่เปิด Messaging API, เอกสาร/FAQ, use-case ชัด และงบ ราคาเริ่ม 60,000 บาท ใช้เวลา 2-6 สัปดาห์ ข้อควรระวังหลักคือ hallucination (คุมด้วย RAG + guardrails), PDPA และต้นทุน token ทำกับทีมที่ออกแบบ RAG เป็น จะได้บอทที่ลูกค้าไว้ใจ ไม่ใช่บอทที่สร้างปัญหา
